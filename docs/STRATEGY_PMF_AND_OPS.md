# ETERNAL SIX — Product-Market Fit & 운영 전략 체크리스트

카톡/문자 복붙 AI 파싱과 자동화된 예약 관리가 시장에 안착하기 위해 **기술·비즈니스적으로 반드시 신경 써야 할 포인트**를 정리한 문서입니다.

---

## 1. 1차 타겟 업종 (Product-Market Fit)

고객이 자체 웹사이트 결제보다 **카카오톡, 인스타 DM, 문자**로 비정형 예약을 문의하는 업종을 최우선 타겟으로 둡니다.

| 업종 | 특징 / 파싱 포인트 |
|------|-------------------|
| **공간 대여업** | 파티룸, 렌탈 스튜디오, 공유 오피스 — 날짜·시간·인원·보증금 파싱 최적화 |
| **반려동물 서비스** | 펫시터, 애견 미용, 펫 택시 — 반려동물 특이사항·픽업 메모 파싱 빈도 높음 |
| **교육·클래스** | 소규모 학원, 원데이 클래스, 개인 과외 — 수강생 관리·스케줄링 |
| **뷰티·에스테틱** | 네일, 헤어샵, 왁싱, 눈썹 — **노쇼 방지·블랙리스트** 수요 가장 큼 |
| **주문 제작업** | 레터링 케이크, 수제 간식, 타투 — 디자인 시안 이미지 파싱 + 주문서 텍스트 결합 |

- 랜딩/키워드: `lib/seo-keywords.ts`에 업종별 `industry`·`painPoint`·`benefit`이 이미 구분되어 있음. 위 5개 업종별 캠페인·콘텐츠 강화 시 참고.

## 2. AI 파싱(OpenAI) 고도화 및 UX

서비스의 **핵심 경쟁력**이자 이탈률을 좌우하는 구간입니다.

### 2.1 프롬프트·OCR 최적화

- 고객 문의는 **"낼 2시 3명요"**, **"이번주 금욜 오후 예약되나용"** 등 초성·줄임말·오타가 많음.
- **할 일**: 한국어 구어체·캡처 이미지 텍스트를 정확히 추출하도록 `app/api/parse-booking/route.ts`의 system prompt를 **엣지 케이스별로 튜닝**.
- 이미지 OCR은 `gpt-4o-mini` 사용 중. 정확도 이슈 시 `gpt-4o` 등 상위 모델 옵션 검토.

### 2.2 환각(Hallucination) 대비 UI

- AI가 날짜·결제 금액을 잘못 인식할 수 있음.
- **할 일**: `PasteDropZone`에서 파싱 결과를 **사장님이 원클릭 검수 + 드롭다운/인라인으로 즉시 수정** 가능한 UI 추가.
- **현재**: 추출 결과를 `<dl>`로만 표시하고 "예약 리스트·캘린더에 추가" 버튼만 제공. **수정 불가** → 검수·수정 UX 보강 필요.

### 2.3 API 비용 통제

- 이미지 OCR·텍스트 파싱 호출이 잦으면 토큰 비용이 커짐.
- **할 일**:
  - 토큰 사용량 **모니터링** (로그·메트릭).
  - 요금제(Trial/Starter/Pro)별 **월별 파싱 횟수 제한** 도입해 마진 보호.
- **현재**: `ParseLog`로 파싱 이력은 저장 중. 요금제별 상한 체크 로직은 미구현.

## 3. 결제·구독(토스페이먼츠) 예외 처리

SaaS 현금 흐름을 담당하므로 **보수적 설계**가 필요합니다.

### 3.1 웹훅 멱등성

- **현재 반영됨**: `WebhookEvent` 테이블과 `lib/webhook-toss.ts`에서 `eventKey(orderId:eventType:paymentKey 등)`로 **멱등성 보장**. 이미 처리된 이벤트는 200만 반환하고 재처리하지 않음. `docs/SECURITY.md`에 명시됨.

### 3.2 구독 해지·유예 기간(Grace Period)

- 카드 잔액 부족 등으로 **구독 결제 실패** 시:
  - 접근을 즉시 차단할지, **며칠 유예**를 줄지 정책 수립 필요.
- **일반적 패턴**: 유예 기간 동안 **읽기 전용** — 데이터 조회는 허용, 수정·신규 파싱·결제는 차단.
- **할 일**:
  - `Subscription.status`에 `past_due` 등 이미 존재. `past_due` 시 **유예 일수**와 **읽기 전용 모드** 적용 로직을 API·대시보드에 반영.
  - 결제 실패 시 재시도·알림(이메일/카톡) 정책 정의.

## 4. 마케팅·SEO (200개 랜딩 페이지)

SSG로 키워드별 랜딩을 생성하는 전략은 유효하나, **검색 엔진 페널티**를 피해야 합니다.

### 4.1 키워드 카니발리제이션(중복 콘텐츠) 방지

- "홍대 네일샵 예약관리", "강남 파티룸 예약툴"처럼 **지역/업종만 다르고 나머지가 99% 동일**하면 Googlebot이 스팸으로 간주해 색인 누락할 수 있음.
- **현재**: `lib/seo-keywords.ts`에서 슬러그별로 `industry`·`painPoint`·`benefit`이 다르게 설정되어 있어 기본적인 차별화는 있음.
- **할 일**:
  - 페이지별 **동적 요소** 추가: 업종별 통계, 지역/업종 특화 Pain Point 문구, 소개 블록 순서 변경 등으로 **콘텐츠 유니크성** 강화.
  - 메타 설명·H1이 슬러그마다 확실히 다르게 노출되는지 점검.

### 4.2 무료 진단 퍼널

- `/diagnosis` 리드 수집 시 **긴 폼은 이탈률을 높임**.
- **할 일**: **카카오 소셜 로그인** 등으로 "클릭 한 번"에 리드 남기기 — 진입 장벽 극단적 축소.
- **현재**: `DiagnosisForm` 기반 이메일·유입 경로 입력. 이메일 중복 제한 등은 있을 수 있으나, 카카오 연동은 별도 구현 필요.

## 5. 데이터 보안·법적 이슈 (매우 중요)

B2B2C 구조이므로 **개인정보**에 민감하게 대응해야 합니다.

### 5.1 개인정보 제3자 제공·수탁·해외 전송

- 사장님이 **고객 이름·전화번호(PII)**가 포함된 대화를 ETERNAL SIX에 올리고, 해당 데이터가 **OpenAI 서버**로 전송됨.
- **할 일**: `/privacy`(개인정보처리방침)에 다음을 **명시**:
  - 예약·문의 내용을 **AI 파싱 목적**으로 활용함.
  - **해외(OpenAI 등)로 전송**될 수 있음.
  - 수탁자·제3자 제공 목록에 OpenAI(및 필요 시 토스 등) 포함.

### 5.2 블랙리스트 격리

- 노쇼 **블랙리스트**는 **해당 사장님(Tenant) 계정 내에서만** 동작해야 함. 다른 업장과 블랙리스트를 공유하면 **심각한 법적 제재** 가능.
- **현재**:
  - 블랙리스트는 **클라이언트 전용** `localStorage`(`lib/blacklist.ts`, 키: `o1t_blacklist`)로만 저장됨.
  - **테넌트 간 공유 이슈 없음** (기기·브라우저 단위 = 로그인 사용자 단위).
- **향후 서버 측 블랙리스트 도입 시**:
  - 반드시 **userId(테넌트)** 기준으로 격리. 다른 사용자 블랙리스트에 절대 노출·연동하지 말 것.
  - `docs/SECURITY.md`에 블랙리스트 격리 원칙 명시됨.

| 구독 유예(읽기 전용) | ⚠ 부분 | status `past_due` 존재, 유예 정책·UI 미정립 |

| 진단 퍼널(카카오 원클릭) | ❌ 미구현 | DiagnosisForm 개선 시 고려 |
